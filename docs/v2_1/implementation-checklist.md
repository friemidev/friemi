# v2.1 功能实现清单

## 当前阶段目标

v2.1 的重点是在 v2.0 完成迁移、品牌基础和核心体验修复后，继续提升产品第一印象和长期维护质量。这个版本不优先堆新功能，而是把首页视觉质感、响应式布局边界和移动端 / 网页端样式架构整理清楚。

当前主线：

```text
v2.0 发布稳定 -> 首页形成更高级的品牌第一印象 -> 移动端和网页端布局彻底解耦 -> 后续 UI 迭代更可控
```

## 分支规则

所有功能从 `dev` 拉独立 feature 分支：

```bash
git switch dev
git pull friemi dev
git switch -c feature/功能名
```

完成后 PR 到：

```text
feature/* -> dev
```

v2.1 不建议把首页视觉重构和响应式架构拆分混在同一个 PR。首页重构偏体验与视觉验证；移动端 / 网页端样式分离偏架构和回归风险，应分开评审和预览。

## P0：v2.1 重点体验优化

### 1. 首页样式优化：高星级酒店式品牌质感

建议分支：

```text
feature/v2-1-home-luxury-hotel-redesign
```

需求理解：

首页需要从普通活动工具首页，升级成更有 Friemi 识别度和信任感的第一屏。视觉方向参考高星级酒店官网的克制、高级、清爽和服务感，但不能做成空洞的营销页。用户打开首页后应该马上理解：Friemi 是一个适合发现活动、约朋友、发起组局的精致生活方式工具。

小功能：

- [ ] 重构首页第一屏视觉，参考高星级酒店官网的留白、字体层级、图片质感和低饱和色彩
- [ ] 首页首屏明确展示 Friemi 品牌、活动发现、组局和朋友同行的核心价值
- [ ] 使用高质量真实感图片或品牌资产，不使用廉价渐变、装饰色块或模板感插画
- [ ] 首页 CTA 层级重排，优先突出发现活动、进入组队大厅和发起组局
- [ ] 最近活动 / 最近组局模块使用更高级的卡片密度和排版，不显得杂乱或像后台列表
- [ ] 移动端首页首屏不被大段文案占满，优先看到品牌、核心入口和内容预览
- [ ] 桌面端首页避免大面积空白，内容宽度、图片比例和入口数量更平衡
- [ ] 首页 footer、header 和主体视觉风格保持一致，不出现 v1/v2 风格混搭

验收标准：

- 首页第一屏有明显的高级感、服务感和 Friemi 品牌识别度
- 用户能在 3 秒内理解产品用途：发现活动、约朋友、一起出发
- 移动端 390px 不出现横向滚动条，首屏不被纯文案撑满
- 桌面端 1280px 下布局不空、不散、不像临时模板
- 主要 CTA 触控高度不低于 44px
- 图片、按钮、卡片和文字层级统一，不出现廉价营销页质感

### 2. 移动端 CSS 与网页端 CSS / 布局彻底分离

建议分支：

```text
feature/v2-1-responsive-style-split
```

需求理解：

当前很多页面通过同一套组件和 Tailwind breakpoint 同时处理移动端与网页端，导致移动端优化时容易破坏桌面端，桌面端美化时又会让移动端出现拥挤、错位或空间浪费。v2.1 需要逐步建立更明确的移动端 / 网页端样式边界，必要时连布局结构也分开，而不是只靠一套 DOM 加响应式 class 硬撑。

小功能：

- [ ] 盘点核心页面中移动端和网页端互相牵制最严重的组件：home、activities、lobby、search、detail、profile
- [ ] 制定移动端 / 网页端样式分离规则：哪些只分 CSS，哪些需要拆组件或拆布局容器
- [x] 为首页建立清晰的 mobile layout 和 desktop layout 边界，避免复杂 breakpoint 串联
- [x] 首页活动轮播增加 mobile / desktop density 边界入口，本分支保持原有卡片宽度、字号和视觉参数不变
- [ ] 移动端优先优化首屏密度、底部导航安全区、触控高度和双列卡片宽度
- [ ] 网页端优先优化内容密度、栅格列数、筛选栏横向布局和大屏留白
- [ ] 避免移动端隐藏元素仍影响桌面端布局，或桌面端容器约束反向影响移动端
- [ ] 梳理共享视觉 token，只共享颜色、字体、圆角、阴影等基础语义，不强行共享布局
- [x] 为后续页面重构记录可复用约定，避免每个页面重新发明响应式结构

验收标准：

- 移动端和网页端可以分别调整核心布局，不再频繁互相破坏
- 390px 移动端核心页面无横向滚动条、底部导航不遮挡主要操作
- 1280px 桌面端核心页面无明显大空白、卡片列数和筛选布局合理
- 组件代码中的 breakpoint 复杂度下降，页面结构更容易阅读和维护
- 后续 UI PR 可以清楚说明影响的是 mobile、desktop 还是共享 token

- [x] 建立共享品牌组件：`BrandLockup` 承载横向 logo 组合图，`BrandBackdrop` 承载 desktop / mobile 品牌背景资产
- [x] 分别优化网页端 header 和移动端 header 的 logo 尺寸、留白和按钮关系
- [x] Header、footer、共创页和微信引导页接入正式横向 lockup，不再各自手写 icon + title 拼接

- [x] 替换残留旧风格按钮、卡片、徽章和空态，不让 v1/v2 风格混杂
- [x] 收敛全局语义色：`coral`、`paper`、`sand`、`team-bg`、`team-border` 切到 v2.1 绿 / 奶油 / 珊瑚系统

- [x] 核心控件第一轮统一：搜索按钮、分页、导航进度条、详情页 CTA、分享 / 复制浮层和日期视图控件
- [x] 共享空状态组件接入 v2.1 品牌 icon、绿色行动按钮和更轻的纸面背景
- [x] 共享空状态和错误态接入专用 `friemi-empty-state-mark.png`
- [x] 移动端底部导航完成 v2.1 品牌试点：奶油底、品牌绿选中态、珊瑚主入口
- [x] 梳理移动端底部导航、浮层、弹窗和安全区，保证新视觉不遮挡核心操作
- [x] 统一 loading、默认分享图和 PWA icon 的品牌露出
- [x] 共享错误页组件接入 v2.1 品牌错误态
- [x] 统一公告页和页面级特殊空状态的品牌露出
- [x] 建立全站品牌视觉审计清单，记录哪些页面已完成、哪些页面仍是旧样式

- 网页端和移动端 logo 在主要入口都清楚、统一、不挤压导航

- [x] 检查并优化英语分享标题、Open Graph description、微信分享兜底文案
- [ ] 本地 Clerk 配置恢复后，补跑 390px 运行态截图检查，确认英法按钮、badge、筛选项和导航没有真实截断
- [x] 保留英文代码 key 和 URL slug，不为了 accent 修改路由、数据库 enum 或稳定标识

验收标准：

- 法语界面不再大面积出现缺少 accent 的文本
- 法语长文案在移动端主要入口不破坏布局
- 不修改稳定 URL、数据库字段、source 标识或代码 key
- 中文、英文文案不被误改

### 5. 网页端组件风格统一、移动端卡片设计与第二阶段风格化统一

```text

```

需求理解：

小功能：

- [ ] 审计网页端常用组件：按钮、chip、badge、筛选栏、排序下拉、分页、弹窗、toast、空态和错误态的风格差异
- [ ] 统一网页端组件的圆角、边框、阴影、hover / active / disabled 状态和触控 / 点击反馈
- [x] 统一网页端活动大厅、组队大厅、搜索结果和个人空间里的卡片外框、信息层级和操作按钮
- [ ] 优化网页端筛选与排序组件，减少“框框堆叠”和临时感，保持信息密度但不拥挤

- [x] 重做移动端组局卡片设计：报名人数、成员头像、状态、发起人、报名 / 再来一局入口在双列布局下不撑高、不出现难看的省略号
- [x] 落实移动端组局 action sheet：卡片内只保留短操作入口，点击后用底部浮层展示报名加入 / 再来一局，避免按钮撑高卡片
- [x] 消息页风格融入：消息列表、好友列表、聊天线程、气泡、输入栏和 loading 骨架接入 v2.1 品牌纸面风格，并修复移动端会话区上下导航遮挡
- [x] 明确移动端活动卡片和组局卡片的视觉差异，语义清楚但保持同一套 Friemi 品牌语言
- [x] 建立活动 / 组局卡片 v2.1 视觉素材板，放入品牌素材组，作为后续组件落地参考
- [ ] 检查中 / 英 / 法长文案在移动端卡片、badge、按钮和筛选控件里的换行与截断策略
- [x] 建立第二阶段组件风格规则，记录哪些组件已经统一、哪些仍保留旧样式

验收标准：

- 网页端主要组件读起来像同一套产品系统，不再像多个阶段拼接出来的样式
- 390px 移动端活动卡片和组局卡片双列稳定，按钮、badge、收藏和状态不互相遮挡
- 移动端组局卡片不会因为报名 / 再来一局 / 英法文案而异常增高
- 网页端 1280px 下卡片、筛选、分页和弹窗风格统一，信息密度合理

### 6. 色盘驱动的全站颜色统一与全背景 Logo 场景化使用

建议分支：

```text
feature/v2-1-brand-palette-token-unification
```

需求理解：

`public` 已新增 `全背景logo.png` 和 `色盘.png`。后续需要以 `色盘.png` 作为全站唯一颜色来源，把当前散落在组件里的旧色值、临时色值和重复语义色收敛成品牌 token；同时评估 `全背景logo.png` 这种方形带底 logo 在 loading fallback、空状态、分享图、无图卡片和 PWA 场景中是否比透明 logo 更合适。

小功能：

- [ ] 从 `apps/web/public/色盘.png` 提取正式品牌色值，定义主色、辅助色、文字色、背景色、边框色、状态色和阴影色 token
- [ ] 盘点全站 hardcoded 色值，区分必须替换、暂时保留和第三方 / 语义例外
- [ ] 将按钮、卡片、badge、筛选器、下拉栏、分页、弹窗、toast、空态和错误态切到统一色盘
- [ ] 统一移动端底部导航、顶部 header、浮层、action sheet 和安全区背景色，避免新旧色混用
- [ ] 评估 `apps/web/public/全背景logo.png` 的适用场景：方形图标、无图 fallback、空状态、loading reduced-motion、分享图或 PWA 入口
- [ ] 保留透明 logo 和横向 lockup 的使用边界，避免所有场景都强行使用同一张图
- [ ] 更新品牌视觉文档，记录色盘 token、使用规则和禁止继续新增散落色值的约定

验收标准：

- 主要页面不再出现明显的新旧颜色混搭
- 全站新增颜色优先来自 `色盘.png` 派生 token，不继续散落硬编码
- 方形全背景 logo 只出现在更合适的场景，不挤压 header 或替代横向品牌露出
- 移动端 390px 和桌面端 1280px 下按钮、卡片、下拉、弹窗颜色一致且可读

### 7. 组局详情页报名 CTA 优先级重构

建议分支：

```text
feature/v2-1-team-detail-cta-redesign
```

需求理解：

当前组局详情页的报名入口在网页端和移动端都不够明显，用户进入详情后容易先浏览信息但找不到核心操作。这个任务需要把组局详情的“报名 / 联系 / 登录 / 游客报名”操作单独重构，并且桌面端和移动端分开设计：网页端提高报名按钮的视觉优先级和固定可见性；移动端使用更轻、更灵动的悬浮圆形操作入口，点击后再展开操作浮窗，避免详情页第一屏被大按钮压住。

小功能：

- [ ] 桌面端组局详情页重新安排右侧或首屏 CTA 区，报名按钮成为最高优先级操作
- [ ] 桌面端报名、联系发起人、复制 / 分享、管理等操作层级清楚，避免多个按钮同权重堆叠
- [ ] 移动端新增悬浮圆形 CTA 图标按钮，避开底部导航和安全区，不遮挡标题、时间、地点和报名状态
- [ ] 移动端点击悬浮按钮后弹出带轻量动效的操作浮窗 / action sheet
- [ ] 已登录用户操作：报名加入、联系发起人；根据状态展示已报名、待审核、满员、已结束等降级态
- [ ] 未登录用户操作：登录报名、游客报名、联系发起人，并明确游客报名和登录报名的视觉层级
- [ ] 浮窗顶部显示当前组局标题或短标题，避免用户误操作到其他组局
- [ ] 中 / 英 / 法长文案在移动端浮窗按钮里不出现难看的省略号或文字切割
- [ ] 组局发起人 / 管理人视角不展示普通报名 CTA，改为管理、查看报名、联系参与者等操作

验收标准：

- 用户进入组局详情页后能立刻识别报名是主操作
- 桌面端 1280px 下报名按钮足够突出，但不破坏详情页信息阅读
- 移动端 390px 下悬浮 CTA 不遮挡底部导航、内容文本或弹窗按钮
- 已登录、未登录、游客报名、已报名、满员、已结束、自己发起等状态都有清楚操作
- 操作浮窗动效轻量，不拖慢页面，也不会让用户误以为页面跳转失败
