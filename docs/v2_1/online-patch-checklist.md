# v2.1 线上补丁清单

## 当前补丁目标

这次补丁只处理 v2.1 上线后已经暴露、会直接影响用户理解或操作的体验问题，不扩大成新一轮大重构。

当前主线：

```text
线上问题快速定位 -> 小范围修复 -> 预览验证 -> 合并 dev -> 生产发布
```

建议分支：

```text
hotfix/v2-1-online-patch
```

## P0：必须修复

### 1. 发现活动排序：时间段活动按“距离今天最近”排序

需求理解：

发现活动里的“从近到远”排序不能只按 `startAt` 绝对时间排。对于跨天或时间段活动，如果活动已经开始但仍未结束，应按这个时间段距离当前日期 / 当前时间的接近程度排序。

例子：

```text
今天是 6 月 30 日
活动 A：6 月 29 日 - 7 月 1 日
活动 B：7 月 1 日开始
```

这两类都应该被视为“离现在很近”，不能因为 A 的开始时间已经过去就被压到很后面。未开始和进行中不应该在排序优先级上互相打架。

小功能：

- [ ] 梳理活动大厅、搜索结果、mobile-home 推荐流中是否复用同一套活动排序逻辑
- [ ] 为时间段活动计算有效排序时间：当前时间落在 `startAt` 和 `endAt` 之间时，排序优先级按“现在 / 今天附近”处理
- [ ] 未开始活动继续按开始时间从近到远
- [ ] 已结束活动仍然低优先级，除非用户显式显示已结束
- [ ] 避免只修 UI 层，后端查询、分页和客户端排序需要保持一致

验收标准：

- 进行中的跨天活动不会因为 `startAt` 已经过了而被排到很后面
- 6 月 29 日 - 7 月 1 日这类活动在 6 月 30 日能出现在近期结果
- 未开始、进行中、时间段活动混排时，用户看到的是“最近相关”而不是“开始时间最早”
- URL query、分页和筛选状态不被破坏

### 2. 通知 / 消息提醒颜色可读性

需求理解：

移动端 header 的通知提醒当前是白色或过浅颜色，在浅色 header / 复杂背景上不明显，用户很难看到有未读消息。需要改成和 v2.1 品牌色协调、但足够醒目的样式。

小功能：

- [ ] 检查 header 通知图标、未读红点、未读数量 badge 在移动端和网页端的颜色
- [ ] 未读状态使用清楚的品牌强调色，不使用白色作为主要提醒色
- [ ] 通知图标 hover / active / focus 状态保持可读
- [ ] iOS 微信浏览器、Safari、Chrome 下 badge 不被头像、语言按钮或 header 边界遮挡

验收标准：

- 移动端未读通知一眼可见
- badge 颜色和 v2.1 品牌系统协调，不像系统默认红点硬贴上去
- 390px 下不遮挡、不溢出、不被白底吞掉

### 3. “报名后显示”信息需要让用户提前知道

需求理解：

当前活动详情里隐藏地址 / 微信 / 联系方式时，用户只看到空白或不够明确的字段，无法理解“报名后可以看到”。需要在详情页明确展示哪些信息报名后可见，降低误解。

小功能：

- [ ] 对 `仅参与者可见详细地址` 的活动，详情页展示明确状态：报名后显示完整地址
- [ ] 对线上活动网址，未报名时展示：报名后显示链接 / 报名通过后显示链接
- [ ] 对需要审核的活动，区分“报名后显示”和“审核通过后显示”
- [ ] 文案保持产品化，不出现功能说明书式长句
- [ ] 复制按钮在无权限时不展示或置为清楚的不可用状态，避免用户以为复制失败

验收标准：

- 用户能在报名前知道哪些信息当前被保护
- 自动确认活动和需审核活动的可见条件不同且清楚
- 详情页字段不再出现“网址：”后面空着、但旁边仍有复制按钮的尴尬状态

### 4. 线上活动与网址字段的展示和操作

需求理解：

当前线上活动 / 网址类地点会出现城市、地址、网址字段语义混乱的问题。例如发布时把网址填在集合地址里，详情页却显示成普通地点或空网址。需要给线上活动一套清楚的展示与兜底规则。

小功能：

- [ ] 发布 / 编辑页支持把线上活动的网址作为明确字段处理，不和普通集合地址混淆
- [ ] 如果地点字段看起来是 URL，详情页优先按线上链接展示，而不是地图地址
- [ ] 线上活动不请求线下天气和地图，避免显示“暂时无法获取活动地天气”
- [ ] URL 展示支持复制和打开链接，移动端不挤压字段
- [ ] 报名后可见的网址需要遵守权限，不提前泄露完整链接
- [ ] 缺少网址时展示清楚兜底，不出现空字段 + 可复制图标

验收标准：

- 线上活动详情页能明确显示“网址 / 线上链接”
- 用户知道什么时候可以看到完整链接
- 普通线下活动不受影响，地图、天气、地点复制仍正常

### 5. 日期时间弹窗默认选中当天

需求理解：

发布 / 编辑组局时，时间输入必须先点日期才能正常录入，体验很别扭。打开日期时间弹窗时，如果当前字段没有日期，应默认选中当天，让用户可以直接改时间并确认。

小功能：

- [ ] 开始时间弹窗打开时，无已有值则默认今天
- [ ] 结束时间弹窗打开时，无已有值则优先参考开始日期；开始日期也没有时默认今天
- [ ] 默认值只用于弹窗交互，不应在用户未确认前静默写入表单
- [ ] 再来一局场景中，开始时间和结束时间清空后仍能正常打开弹窗并选择
- [ ] 移动端和网页端日期时间弹窗行为一致

验收标准：

- 用户打开时间弹窗后可以直接调整时间，不需要先点一次日期
- 没有确认前不会误保存今天日期
- 再来一局、编辑活动、新建活动都不出现时间输入卡住的问题

## P1：移动端大厅补丁

### 6. mobile-home 分类入口与推荐卡片顺序调整

需求理解：

移动端大厅当前推荐 swiper 卡片压在最上方，分类九宫格被放到下方，用户打开后不够像“大厅入口”。这次补丁把分类入口提到上方并放大，推荐 swiper 下移作为补充内容；同时去掉分类插画背后的明显小框，让九宫格更轻、更像应用首页入口。

小功能：

- [x] 移动端 `/mobile-home` 分类九宫格上移到推荐 swiper 之前
- [x] 分类插画尺寸放大，提升入口辨识度
- [x] 去掉分类插画背后的明显小框和晕染，只保留插画、文字胶囊和轻量动效
- [x] 分类入口改用 SVG 插画源，放大后不糊
- [x] 分类文字压进插画内部展示，三语文案随当前 locale 渲染，不单独维护 27 张图片
- [x] 针对 390px 以下小屏 / 极小屏单独放大分类入口并压缩间距
- [x] 放大图内文字胶囊，小机型减少分类区和下方搜索区的距离
- [x] 推荐 swiper 卡片下移，作为下方推荐区
- [ ] 检查 iPhone SE、iPhone 14 Pro Max 等高度差异下是否仍然不遮挡底部导航

验收标准：

- 移动端首屏优先看到分类入口
- 九宫格插画不再像被小卡片框住
- 分类文字不再单独占一行，图形整体更大
- 推荐 swiper 不再占据最顶部主视觉
- 分类、搜索、推荐卡片和底部导航之间没有明显挤压或遮挡

## 验收顺序

- [ ] 本地手动验证发布 / 编辑时间弹窗
- [ ] 本地手动验证线上网址活动详情页
- [ ] 本地手动验证需审核活动的报名后显示状态
- [ ] 本地手动验证通知未读 badge
- [ ] 本地手动验证移动端大厅分类入口和推荐卡片顺序
- [ ] 预览环境验证活动排序和分页
- [ ] 生产发布后用真实线上链接回归一次通知、详情页和活动大厅

## 非目标

- 不重做活动详情页整体布局
- 不新增复杂线上活动类型系统
- 不把通知中心整页重新设计
- 不重写活动大厅筛选器
- 不调整 v2.1 之外的长期规划功能
