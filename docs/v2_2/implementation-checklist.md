# v2.2 功能实现清单

## 当前阶段目标

v2.2 开始从 v2.1 的视觉和移动端体验整理，进入更深的协作、运营和留存能力建设。重点不是一次性堆满所有功能，而是把组局协作权限、消息表达、签到、冷门分类补全和用户成长体系拆成可独立验证的分支。

当前主线：

```text
v2.1 体验稳定 -> 组局协作能力增强 -> 活动现场和长期留存能力补齐 -> 用户在平台内有更多可持续互动
```

## 分支规则

所有功能从 `dev` 拉独立 feature 分支：

```bash
git switch dev
git pull friemi dev
git switch -c feature/功能名
```

完成后 PR 到：

```text
feature/* -> dev
```

v2.2 不建议把组局权限、聊天图片、签到和成就系统放在同一个 PR。涉及数据库结构、权限边界或用户通知的功能必须单独分支开发，方便预览、回滚和生产迁移。

## P0：协作权限与核心体验

### 1. 组局管理人

建议切换分支：

```text
feature/v2-team-co-managers
```

需求理解：

组局发起人需要能指定最多 3 位管理人协助运营组局。管理人必须是发起人的好友；被指定后可以处理邀请 / 报名审核、修改组局信息、取消组局等管理动作，降低长期组局或大型组局的组织压力。

小功能：

- [x] 设计组局管理人数据模型：活动 / 组局与管理人 Profile 的关联关系
- [x] 限制每个组局最多 3 位管理人
- [x] 只能从发起人的好友中选择管理人
- [x] 发起人可以新增、移除管理人，并看到当前管理人列表
- [x] 管理人可以处理报名 / 邀请审核
- [x] 管理人可以修改组局关键信息，但敏感操作需要权限边界清楚
- [x] 管理人可以取消组局，取消动作需要二次确认和日志记录
- [x] 详情页、管理面板和通知中区分发起人与管理人身份
- [x] 权限校验必须在服务端 action / query 层完成，不能只靠前端隐藏按钮

验收标准：

- [x] 发起人只能选择自己的好友作为管理人
- [x] 第 4 位管理人无法添加，并给出清楚提示
- [x] 管理人能进入对应组局管理入口并完成审核、修改和取消等授权动作
- [x] 非管理人不能通过直接请求调用管理操作
- [x] 发起人移除管理人后，对方立即失去管理权限

### 2. 聊天功能支持图片和表情包

建议切换分支：

```text
feature/v2-chat-media-messages
```

需求理解：

当前消息只能发送文字，组局沟通时缺少图片、截图、活动海报和表情包表达。v2.2 需要扩展聊天消息类型，支持图片上传、图片预览和常用表情包入口，同时保证存储、大小限制和安全展示可控。

小功能：

- [ ] 设计消息媒体数据模型：文字、图片、表情包的消息类型和附件字段
- [ ] 图片上传走受控存储 bucket，限制格式、大小和数量
- [ ] 聊天列表和会话详情支持图片缩略图、点击预览和加载失败兜底
- [ ] 增加轻量表情包入口，不影响文字输入效率
- [ ] 移动端图片发送、预览和键盘弹起时不遮挡输入框
- [ ] 服务端校验会话成员权限，不能仅靠前端隐藏上传入口

验收标准：

- [ ] 用户能在私聊 / 组局相关聊天中发送图片
- [ ] 图片消息在移动端和网页端都能正常预览
- [ ] 非会话成员不能通过接口上传或读取消息图片
- [ ] 大图、坏图和上传失败都有明确反馈

### 3. 冷门分类补全与随机滑一滑组件

建议切换分支：

```text
feature/v2-category-fallback-swipe
```

需求理解：

九宫格分类统一后，部分冷门分类可能只有少量活动甚至没有有效卡片。用户点击分类后不能只看到空白页，需要用“随机滑一滑”补充推荐内容，让探索体验保持连续。

小功能：

- [ ] 分类结果为空或过少时展示随机滑一滑组件
- [ ] 随机内容需要去重，避免刚看过的卡片重复出现
- [ ] 明确区分“当前分类结果”和“为你随机推荐”
- [ ] 移动端组件高度紧凑，不挤掉主要分类信息
- [ ] 桌面端有自然的横向推荐布局，不显得像错误态

验收标准：

- [ ] 冷门分类没有内容时不会出现空白死路
- [ ] 用户仍能通过滑一滑继续发现活动或组局
- [ ] 推荐内容不破坏分类筛选的语义

### 4. 活动头像弹窗内直接关注

建议切换分支：

```text
feature/v2-profile-popover-follow-action
```

需求理解：

当前活动头像弹窗里的关注动作不能立即完成，需要跳到个人主页才可以关注，打断浏览活动的上下文。v2.2 需要让用户在活动详情、卡片头像或参与者弹窗中直接关注 / 取消关注。

小功能：

- [ ] 活动头像弹窗接入关注 / 取消关注 action
- [ ] 操作后弹窗内状态即时更新，不强制跳转主页
- [ ] 未登录用户点击关注时进入登录引导
- [ ] 关注按钮在中文、英文、法语下不挤压弹窗布局
- [ ] 服务端校验不能关注自己、不能重复关注

验收标准：

- [ ] 用户能在头像弹窗内直接关注别人
- [ ] 操作完成后粉丝数和按钮状态同步变化
- [ ] 失败时保留当前弹窗并给出清楚提示

## P1：活动现场和工具能力

### 5. 桌游小工具：狼人杀

建议切换分支：

```text
feature/v2-board-game-werewolf-tool
```

需求理解：

桌游类组局需要轻量现场辅助工具。狼人杀工具先作为独立小工具试点，帮助组织者快速分配角色、管理夜晚流程和记录状态，降低现场主持成本。

小功能：

- [ ] 新建狼人杀工具入口，可从桌游分类或组局管理入口进入
- [ ] 支持人数、角色池和角色分配
- [ ] 支持主持人视角查看角色，不向普通参与者泄露
- [ ] 支持夜晚 / 白天流程提示和基础记录
- [ ] 移动端现场使用优先，按钮触控和字号适合单手操作

验收标准：

- [ ] 主持人能在手机上完成一局基础狼人杀的角色分配和流程推进
- [ ] 普通用户不能看到不该看的角色信息
- [ ] 工具不影响原有组局报名和聊天流程

### 6. 线下活动签到功能

建议切换分支：

```text
feature/v2-offline-check-in
```

需求理解：

线下活动需要知道谁实际到场，方便发起人统计、复盘、后续联系和成就系统沉淀。签到功能要支持现场快速操作，也要考虑二维码、手动签到和迟到补签等场景。

小功能：

- [ ] 设计签到数据模型：活动、参与者、签到状态、签到时间和操作者
- [ ] 发起人 / 管理人可以查看签到名单并手动签到
- [ ] 支持签到二维码或签到码方案，降低现场手工操作
- [ ] 区分已报名、待审核、已到场、未到场和补签状态
- [ ] 签到记录写入日志，避免误操作无法追溯
- [ ] 移动端签到页适合现场快速连续操作

验收标准：

- [ ] 发起人和管理人能完成现场签到
- [ ] 普通参与者不能伪造他人签到
- [ ] 签到数据可用于后续周报、月报和成就统计

## P2：长期留存与成长体系

### 7. 用户称号、成就、周报和月报系统

建议切换分支：

```text
feature/v2-user-achievements-reports
```

需求理解：

用户在 Friemi 中持续参加活动、发起组局、结识朋友后，需要被平台记录和反馈。称号、成就、周报和月报可以把零散行为转化为成长感和回忆感，提升用户留存和分享意愿。

小功能：

- [ ] 设计称号与成就体系：参与、发起、连续活跃、分类偏好、社交关系等维度
- [ ] 用户主页展示部分称号和成就，不干扰主要资料信息
- [ ] 周报展示本周参加 / 发起 / 收藏 / 认识的人等轻量总结
- [ ] 月报展示更完整的活动足迹和偏好变化
- [ ] 支持分享周报 / 月报为图片或链接
- [ ] 成就计算需要可重复执行，避免数据重复或遗漏
- [ ] 隐私设置明确，用户可选择不展示或不分享

验收标准：

- [ ] 用户能在个人空间看到自己的称号和部分成就
- [ ] 周报 / 月报数据与实际报名、签到和发起记录一致
- [ ] 分享内容不泄露用户隐私或他人敏感信息
