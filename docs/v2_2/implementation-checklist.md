# v2.2 功能实现清单

## 当前阶段目标

v2.2 开始从 v2.1 的视觉和移动端体验整理，进入更深的协作、运营和留存能力建设。重点不是一次性堆满所有功能，而是把组局协作权限、消息表达、签到、冷门分类补全和用户成长体系拆成可独立验证的分支。

当前主线：

```text
v2.1 体验稳定 -> 组局协作能力增强 -> 活动现场和长期留存能力补齐 -> 用户在平台内有更多可持续互动
```

## 分支规则

所有功能从 `dev` 拉独立 feature 分支：

```bash
git switch dev
git pull friemi dev
git switch -c feature/功能名
```

完成后 PR 到：

```text
feature/* -> dev
```

v2.2 不建议把组局权限、聊天图片、签到和成就系统放在同一个 PR。涉及数据库结构、权限边界或用户通知的功能必须单独分支开发，方便预览、回滚和生产迁移。

分支合并建议：

- 聊天图片和表情包属于同一消息能力，建议放在 `feature/v2-chat-media-messages`
- 公共活动 API 导入比例和 Swiper 展示组件可以先分开做；如果只做前端展示，不碰导入策略，可合并到 `feature/v2-discovery-swiper-components`
- 隐藏关注 / 粉丝展示和 README 重写都相对独立，不建议和数据库迁移、聊天媒体或签到混在一起
- 组局发布缺字段引导属于发布页核心体验，建议单独 PR，避免和表单业务字段改动互相影响
- Android 移动端 APP 先做规划文档，不在同一分支直接引入移动端工程基础设施

## P0：协作权限与核心体验

### 1. 组局管理人

建议切换分支：

```text
feature/v2-team-co-managers
```

需求理解：

组局发起人需要能指定最多 3 位管理人协助运营组局。管理人必须是发起人的好友；被指定后可以处理邀请 / 报名审核、修改组局信息、取消组局等管理动作，降低长期组局或大型组局的组织压力。

小功能：

- [x] 设计组局管理人数据模型：活动 / 组局与管理人 Profile 的关联关系
- [x] 限制每个组局最多 3 位管理人
- [x] 只能从发起人的好友中选择管理人
- [x] 发起人可以新增、移除管理人，并看到当前管理人列表
- [x] 管理人可以处理报名 / 邀请审核
- [x] 管理人可以修改组局关键信息，但敏感操作需要权限边界清楚
- [x] 管理人可以取消组局，取消动作需要二次确认和日志记录
- [x] 详情页、管理面板和通知中区分发起人与管理人身份
- [x] 权限校验必须在服务端 action / query 层完成，不能只靠前端隐藏按钮

验收标准：

- [x] 发起人只能选择自己的好友作为管理人
- [x] 第 4 位管理人无法添加，并给出清楚提示
- [x] 管理人能进入对应组局管理入口并完成审核、修改和取消等授权动作
- [x] 非管理人不能通过直接请求调用管理操作
- [x] 发起人移除管理人后，对方立即失去管理权限

### 2. 聊天功能支持图片和表情包

建议切换分支：

```text
feature/v2-chat-media-messages
```

需求理解：

当前消息只能发送文字，组局沟通时缺少图片、截图、活动海报和表情包表达。v2.2 需要扩展聊天消息类型，支持图片上传、图片预览和常用表情包入口，同时保证存储、大小限制和安全展示可控。

小功能：

- [ ] 设计消息媒体数据模型：文字、图片、表情包的消息类型和附件字段
- [ ] 图片上传走受控存储 bucket，限制格式、大小和数量
- [ ] 修复当前发消息不能发图片的问题，先支持基础图片消息发送、预览和失败重试
- [ ] 聊天列表和会话详情支持图片缩略图、点击预览和加载失败兜底
- [ ] 增加轻量表情包入口，可先用一组内置表情包 / sticker，后续再扩展收藏和搜索
- [ ] 移动端图片发送、预览和键盘弹起时不遮挡输入框
- [ ] 服务端校验会话成员权限，不能仅靠前端隐藏上传入口

验收标准：

- [ ] 用户能在私聊 / 组局相关聊天中发送图片
- [ ] 图片消息在移动端和网页端都能正常预览
- [ ] 非会话成员不能通过接口上传或读取消息图片
- [ ] 大图、坏图和上传失败都有明确反馈

### 3. 冷门分类补全与随机滑一滑组件

建议切换分支：

```text
feature/v2-category-fallback-swipe
```

需求理解：

九宫格分类统一后，部分冷门分类可能只有少量活动甚至没有有效卡片。用户点击分类后不能只看到空白页，需要用“随机滑一滑”补充推荐内容，让探索体验保持连续。

小功能：

- [ ] 分类结果为空或过少时展示随机滑一滑组件
- [ ] 随机内容需要去重，避免刚看过的卡片重复出现
- [ ] 明确区分“当前分类结果”和“为你随机推荐”
- [ ] 移动端组件高度紧凑，不挤掉主要分类信息
- [ ] 桌面端有自然的横向推荐布局，不显得像错误态

验收标准：

- [ ] 冷门分类没有内容时不会出现空白死路
- [ ] 用户仍能通过滑一滑继续发现活动或组局
- [ ] 推荐内容不破坏分类筛选的语义

### 4. 活动头像弹窗内直接关注（暂缓）

建议切换分支：

```text
feature/v2-profile-popover-follow-action
```

需求理解：

当前活动头像弹窗里的关注动作不能立即完成，需要跳到个人主页才可以关注，打断浏览活动的上下文。v2.2 需要让用户在活动详情、卡片头像或参与者弹窗中直接关注 / 取消关注。

当前状态：

由于 v2.2 新增“隐藏全站关注和粉丝展示入口”，本项暂缓进入近期开发。等关注 / 粉丝关系重新对用户开放时，再恢复本项。

小功能：

- [ ] 活动头像弹窗接入关注 / 取消关注 action
- [ ] 操作后弹窗内状态即时更新，不强制跳转主页
- [ ] 未登录用户点击关注时进入登录引导
- [ ] 关注按钮在中文、英文、法语下不挤压弹窗布局
- [ ] 服务端校验不能关注自己、不能重复关注

验收标准：

- [ ] 用户能在头像弹窗内直接关注别人
- [ ] 操作完成后粉丝数和按钮状态同步变化
- [ ] 失败时保留当前弹窗并给出清楚提示

### 5. 组局发布缺字段引导和步骤错误提示

建议切换分支：

```text
feature/v2-team-publish-validation-guidance
```

需求理解：

组局发布页如果某个必填字段没填，现在用户只知道“发不出去”，但不一定能快速知道是哪一步、哪个字段出了问题。v2.2 需要把表单校验反馈前移到顶部步骤导航和字段本身，让用户能一眼定位问题。

小功能：

- [x] 顶部 4 个步骤按钮接入校验状态：对应步骤有缺字段时显示醒目的感叹号标志
- [x] 点击带感叹号的步骤后，滚动 / 切换到对应步骤并聚焦第一个错误字段
- [x] 字段错误提示写在字段附近，不只依赖 toast 或提交后的通用报错
- [x] 多语言下错误提示不挤压移动端布局
- [x] 提交失败后保留用户已填写内容，不清空表单
- [x] 长期活动、报名审核、费用、地点隐私等特殊字段也纳入校验提示

验收标准：

- [x] 用户提交失败后能立刻知道是哪一步缺字段
- [x] 顶部步骤状态、字段边框和错误文案三者一致
- [x] 390px 移动端下错误提示不遮挡底部操作按钮

### 6. 公共活动 API 导入质量调优

建议切换分支：

```text
feature/v2-public-activity-import-quality-tuning
```

需求理解：

当前 API 导入的公共活动里长期活动占比偏高，容易让发现页显得不够“今天可以去”。v2.2 需要优化导入和排序策略，减少长期活动比例，增加单日 / 短周期活动，并适当延长前瞻周期，让内容更丰富但不失去时效感。

小功能：

- [x] 统计导入结果中长期活动、单日活动、短周期活动的比例
- [x] 调整导入选择策略，降低长期活动占比，提高单日和短周期活动比例
- [x] 前瞻周期适当延长，但避免一次导入大量低相关远期内容
- [x] 对同一来源、同一标题、同一日期范围继续做去重和更新
- [x] 导入日志记录本次导入的分类、周期和过滤原因，方便后续调参
- [x] 发现活动排序继续优先展示近期可参加内容，长期活动不长期霸占首屏

验收标准：

- [x] 发现页首屏有更多单日 / 短周期活动
- [x] 长期活动仍可被发现，但比例更克制
- [x] cron dry-run 可以看出导入比例和过滤结果

### 7. 隐藏全站关注和粉丝展示入口

建议切换分支：

```text
feature/v2-hide-follow-surfaces
```

需求理解：

关注 / 粉丝关系目前还不是 v2.2 想重点对外展示的社交逻辑。为了降低新用户理解成本，可以先隐藏全站关注、粉丝按钮和显性数字，只保留底层数据与好友等更核心关系，等社交体系更完整后再重新开放。

小功能：

- [x] 隐藏个人主页中的关注 / 粉丝入口和显性计数
- [x] 隐藏头像弹窗、参与者弹窗和卡片中的关注按钮
- [x] 保留好友、私聊、参与记录等更直接的关系入口
- [x] 不删除数据库字段和后端能力，避免破坏历史关系数据
- [x] 中 / 英 / 法文案同步调整，不再提示用户关注 / 粉丝逻辑

验收标准：

- [x] 普通用户浏览主路径时看不到关注 / 粉丝按钮和入口
- [x] 现有关注数据不被删除，未来可恢复展示
- [x] 隐藏后布局不出现空洞或按钮错位

### 8. Swiper 滑动发现组件复用

建议切换分支：

```text
feature/v2-discovery-swiper-components
```

需求理解：

首页、大厅、冷门分类补全和活动推荐都需要更轻量的滑动发现体验。v2.2 可以把已经验证过的 swiper 滑动卡片整理成可复用组件，在合适的位置安排上，但避免所有页面都堆同一种模块。

小功能：

- [ ] 抽象活动 / 组局 swiper 组件，支持标题、数据源、空状态和自动 / 手动切换
- [ ] 冷门分类结果不足时接入 swiper 推荐，避免空白页
- [ ] 大厅或移动端入口根据内容密度安排滑动推荐，不重复占用首屏
- [ ] 支持去重、分页加载和快到底时补下一批
- [ ] 移动端手滑不重置进度，桌面端左右按钮和键盘操作可用
- [ ] 减少 motion 对性能弱设备的影响

验收标准：

- [ ] swiper 不会滑到尽头后变成死路
- [ ] 推荐内容和当前页面语义清楚，不误导用户
- [ ] 移动端滑动自然，不横向撑爆页面
