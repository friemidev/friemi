# 狼人杀线下辅助工具 MVP 开发规划与 Checklist

> 文档版本：v0.1
> 生成日期：2026-07-06
> 适用阶段：v2.3 产品范围确认、MVP 研发拆解、桌游工具平台扩展
> 产品形态：移动 Web / WebView 优先，适配浏览器、Android 壳、后续 iOS 壳
> 设计原则：工具辅助线下主持，不替代玩家发言、推理和法官裁量

---

## 0. v2.3 定位

狼人杀工具是 v2.3 桌游工具线的重点功能之一。它不要求等待原生 App；当前 Friemi 的移动 Web 和 Android WebView 壳已经足够承载第一版狼人杀工具。

v2.3 的合理策略是：

```text
移动 Web 先做好 App 化体验
狼人杀工具优先 Web 实现
Android 壳直接复用
iOS 壳后续复用同一套移动 Web
原生化等到语音、房间实时通信、复杂媒体流成为主线后再评估
```

狼人杀 MVP 的目标不是做完整线上狼人杀，也不是自动主持整局游戏，而是作为线下法官的辅助工具，解决线下桌游最实际的问题：

- 快速建房
- 玩家和法官入座
- 自动随机发身份
- 玩家安全查看自己的身份
- 法官查看全局身份和存活状态
- 法官标记死亡 / 复活
- 法官手动结束游戏并记录胜负
- 将结果沉淀到个人记录

产品边界必须清楚：

```text
系统负责：建房、入座、发身份、防窥查看、死亡状态、手动结算、个人记录
线下法官负责：发言、投票、计时、夜晚行动、用药验人、流程节奏、胜负裁量
```

这意味着狼人杀工具不追求“线上完整游戏房间”。玩家仍然围坐在线下，法官继续口述和裁判，Friemi 只把容易出错、容易泄露、需要沉淀记录的部分数字化。

## 1. 与现有桌游工具的关系

v2.2 已经通过 Avalon 沉淀出一套可复用底座：

- 统一桌游工具大厅 `/game-tools`
- `GameToolRoom` 房间
- `GameToolSeat` 座位和私密 token
- `GameToolEvent` 事件日志
- 短房号、私密链接、revalidate 工具
- 桌游组局详情页的工具入口

狼人杀应复用这套底座，而不是另起一套房间系统。

需要新增或调整：

- `GameToolKind.WEREWOLF`
- 狼人杀版型配置
- 狼人杀房间状态机
- 狼人杀角色配置与随机发牌
- 法官位
- 房间成员区
- 玩家角色卡牌页
- 法官死亡标记与结束游戏
- 玩家游戏记录 / 战绩统计

## 2. MVP 范围

MVP 阶段只上线：

```text
10 人预女猎局 = 9 名玩家 + 1 名法官
```

发牌对象只有 9 名玩家，法官不参与发牌。

9 名玩家身份配置：

```text
狼人 x 3
预言家 x 1
女巫 x 1
猎人 x 1
平民 x 3
```

阵营配置：

```text
狼人阵营：狼人
好人阵营：预言家、女巫、猎人、平民
法官：不属于任一阵营，不参与胜负
```

MVP 不做：

- 不自动控制夜晚流程
- 不自动判断刀人、验人、用药和开枪
- 不做系统内发言计时器
- 不做系统内投票流程或票型统计
- 不做语音聊天
- 不做完整线上房间实时语音
- 不做多人文字聊天
- 不做实时游戏房间或线上对局
- 不做复杂房规自定义
- 不做 7 / 8 / 9 / 12 人局完整规则
- 不做 AI 推理、策略建议或胜率预测

## 3. 推荐访问路径

```text
/{locale}/game-tools
/{locale}/game-tools/werewolf
/{locale}/game-tools/werewolf/rooms/{roomId}
/{locale}/game-tools/werewolf/join/{code}
/{locale}/game-tools/werewolf/seats/{privateToken}
```

后续可选：

```text
/{locale}/game-tools/werewolf/rooms/{roomId}/screen
/{locale}/game-tools/werewolf/rooms/{roomId}/recap
```

MVP 阶段可以先不做公共屏和复盘页。

## 4. 整体流程

```text
桌游组局详情页
  -> 桌游工具入口
  -> 桌游工具大厅
  -> 狼人杀
  -> 版型选择
  -> 创建房间
  -> 成员进入房间
  -> 玩家 / 法官选座
  -> 全员准备
  -> 法官开始游戏
  -> 系统随机发身份
  -> 玩家进入角色卡牌页
  -> 法官留在房间页管理状态
  -> 法官标记死亡 / 复活
  -> 法官手动结束游戏
  -> 系统计算玩家胜负
  -> 结算动画
  -> 返回房间
  -> 写入个人记录
```

## 5. 页面设计

### 5.1 桌游工具入口

入口规则：

- 用户创建组局时，如果组局类型选择为「桌游」，组局详情页展示「桌游工具」入口。
- 已报名 / 已通过 / 发起人 / 管理人可看到入口。
- 入口进入统一桌游工具大厅。
- 大厅展示：狼人杀、阿瓦隆、血染钟楼 / 说书人工具。
- MVP 阶段狼人杀显示为可使用，其余工具保持现有状态。

Checklist：

- [x] 在 `gameToolCatalog.ts` 增加狼人杀工具配置
- [x] 桌游工具大厅展示狼人杀卡片
- [ ] 桌游组局详情页的工具入口优先展示狼人杀
- [x] 狼人杀入口文案支持 `zh-CN`、`en`、`fr`
- [ ] Android WebView 中打开入口不出现横向溢出或安全区遮挡

### 5.2 版型选择页

用户点击狼人杀后进入版型选择。

版型从 7 人局开始依次罗列：

- 7 人局
- 8 人局
- 9 人局
- 10 人预女猎局
- 12 人预女猎白痴局

注意：这里的人数包含法官。

MVP 状态：

- 只有「10 人预女猎局」可点击创建
- 其他版型展示为规划中或禁用态

Checklist：

- [x] 新增 `/game-tools/werewolf` 页面
- [x] 展示版型列表和人数说明
- [x] 明确说明人数包含法官
- [x] 10 人预女猎局可创建房间
- [x] 其他版型暂不可选，并展示后续支持提示
- [x] 版型配置写入独立 `werewolfConfig.ts`

用户选择版型后进入狼人杀房间。

房间页面分为两部分：

上半部分：座位区。

10 人预女猎局显示：

- 1 号座位
- 2 号座位
- 3 号座位
- 4 号座位
- 5 号座位
- 6 号座位
- 7 号座位
- 8 号座位
- 9 号座位
- 法官位

下半部分：房间成员区。

- 已进入房间但未选座的用户先显示在成员区。
- 选座后从成员区移动到对应座位。
- 游戏开始后房间锁定，不能继续加入本局游戏。

Checklist：

- [x] 新增 `/game-tools/werewolf/rooms/{roomId}` 页面
- [x] 房间显示短房号和分享链接
- [x] 房间展示 1-9 号玩家座位和法官位
- [x] 房间展示未落座成员区
- [x] 空座位可点击选择
- [x] 已占座位展示头像 / 昵称 / 准备状态
- [x] 游戏开始前允许换座

### 5.4 选座流程

规则：

- 玩家可以选择 1-9 号玩家座位。
- 一名用户可以选择法官位。
- 每个座位只能被一个用户占用。
- 游戏开始前可以换座。
- 游戏开始后所有座位锁定。

Checklist：

- [x] 选座 action 校验房间必须处于候场阶段
- [x] 选座 action 校验目标座位为空
- [x] 换座时释放原座位并占用新座位

- [x] 开始后不能换座
- [x] 非房间成员不能直接通过请求抢座
- [x] 误点换座有清晰反馈

### 5.5 准备流程

所有玩家和法官都落座后，每个人点击「准备」。

当所有人准备后：

- 房间进入可开始状态
- 只有法官可以点击「开始游戏」

点击开始游戏后：

- 房间锁定
- 不能换座
- 不能换法官
- 不能加入本局游戏
- 系统随机分发角色

Checklist：

- [x] 每个座位有独立准备状态
- [x] 玩家和法官都需要准备
- [x] 未满 9 名玩家 + 1 名法官时不能开始
- [x] 全员准备后展示可开始状态
- [x] 只有法官位用户能开始游戏
- [x] 开始游戏需要二次确认
- [x] 开始游戏写入 `GameToolEvent`

### 5.6 游戏开始与发牌

法官点击「开始游戏」后：

- 所有人进入游戏开始动画
- 系统随机分配 9 张角色牌
- 法官不参与发牌
- 玩家座位写入各自私密身份
- 法官页可看到全部角色

Checklist：

- [x] `werewolfConfig.ts` 定义 10 人预女猎局角色池
- [x] 发牌逻辑只对 1-9 号玩家座位执行
- [x] 法官位不写入角色
- [x] 随机洗牌后写入每个座位 `roleKey`、`roleAlignment`、`privatePayload`
- [x] 发牌完成后房间状态进入 `IN_PROGRESS`
- [x] 发牌动作写入事件日志
- [x] 发牌失败时房间不能进入半开始状态

### 5.7 玩家角色卡牌页

游戏开始后，玩家跳转或进入自己的角色卡牌页面：

```text
/{locale}/game-tools/werewolf/seats/{privateToken}
```

页面包含三部分：

1. 座位号：页面上半部分中央显示玩家座位号。
2. 规则说明：显示当前版型、角色功能、阵营和简单规则。
3. 查看角色按钮：点击后卡牌翻转，显示自己的角色。

角色显示规则：

- 默认不展示角色
- 点击「查看我的角色」后翻牌
- 显示 2 秒后自动翻回
- 玩家可以重复点击查看
- 死亡后卡牌变灰并显示「你已出局」

Checklist：

- [x] 私密页通过 `privateToken` 访问
- [x] 私密页只能看到自己的身份
- [x] 页面顶部突出座位号
- [x] 角色卡默认防窥
- [x] 点击后翻牌展示 2 秒
- [x] 2 秒后自动翻回卡背
- [x] 重复点击不会泄露其他身份
- [x] 角色说明为 Friemi 原创简短文案，不复制外部说明书
- [ ] 被标记死亡后展示死亡动画
- [ ] 死亡动画后卡牌变灰并显示「你已出局」

### 5.8 法官页面

游戏开始后，法官继续停留在房间页面。

法官可以看到：

- 所有玩家座位号
- 所有玩家头像
- 所有玩家昵称
- 所有玩家角色缩图
- 所有玩家当前状态

玩家状态包括：

- 存活
- 死亡

MVP 阶段，法官线下口述流程，系统只提供身份和状态辅助：

- 天黑请闭眼
- 狼人请睁眼
- 预言家请睁眼
- 女巫请睁眼
- 天亮了
- 进入发言
- 进入投票
- 宣布死亡

系统不自动控制狼人杀流程，也不记录投票、发言时间、夜晚行动和技能使用。法官按线下规则自行主持和裁量。

Checklist：

- [x] 法官页显示全部角色信息
- [ ] 非法官不能访问全部身份视图
- [ ] 法官页支持存活 / 死亡状态切换
- [ ] 法官页提供简短主持词提示
- [ ] 主持词只是提示，不驱动强制流程
- [ ] 法官页在移动端单手可操作
- [ ] 关键操作需要防误触设计

### 5.9 死亡判定

游戏过程中，如果法官判定某位玩家死亡，法官点击该玩家座位头像。

操作：

- 标记死亡
- 取消死亡

效果：

- 被标记死亡的玩家进入死亡状态
- 玩家角色卡牌页出现死亡动画
- 动画结束后卡牌变灰
- 页面显示「你已出局」
- 玩家不能再参与游戏操作
- 玩家仍可留在房间观看
- 法官页面中该玩家头像变灰并显示死亡状态

Checklist：

- [ ] 法官可标记玩家死亡
- [ ] 法官可取消死亡
- [ ] 死亡 / 复活动作写入事件日志
- [ ] 玩家私密页能反映死亡状态
- [ ] 死亡动画只在状态首次变化时触发
- [ ] 死亡玩家仍能查看房间公开状态
- [ ] 非法官不能通过请求修改死亡状态

### 5.10 游戏结束

MVP 阶段由法官手动结束游戏。

法官点击「结束游戏」后选择获胜阵营：

系统根据玩家所属阵营判断每位玩家本局胜利或失败。

## 6. 推荐数据模型

### 6.1 GameToolKind

新增：

```prisma
enum GameToolKind {
  AVALON
  STORYTELLER
  WEREWOLF
}
```

### 6.2 GameToolRoom.config

狼人杀房间配置建议写入 `GameToolRoom.config`：

```ts
type WerewolfRoomConfig = {
  kind: "WEREWOLF";
  variantKey: "ten_player_seer_witch_hunter";
  variantName: string;
  totalSeats: 10;
  playerSeatCount: 9;
  hasJudge: true;
  roleDeck: WerewolfRoleKey[];
};
```

### 6.3 GameToolRoom.state

狼人杀房间状态建议写入 `GameToolRoom.state`：

```ts
type WerewolfRoomState = {
  phase: "LOBBY" | "READY" | "DEALING" | "IN_PROGRESS" | "FINISHED";
  lockedAt?: string | null;
  startedAt?: string | null;
  finishedAt?: string | null;
  winner?: "GOOD" | "WEREWOLF" | null;
  deadSeatNumbers: number[];
  resultRecordedAt?: string | null;
};
```

### 6.4 GameToolSeat

优先复用现有 `GameToolSeat`：

- 1-9 号为玩家座位
- 10 号或特殊配置显示为法官位
- 玩家座位有 `roleKey` 和 `roleAlignment`
- 法官位 `roleKey = null`
- 每个座位仍有 `privateToken`

法官识别建议不要只靠座位号写死在 UI，应由狼人杀配置判断。

### 6.5 房间成员

为了支持“进入房间但未选座的用户显示在成员区”，建议新增通用成员模型，而不是狼人杀私有模型：

```prisma
model GameToolRoomMember {
  id            String       @id @default(cuid())
  roomId        String
  room          GameToolRoom @relation(fields: [roomId], references: [id], onDelete: Cascade)
  profileId     String?
  guestName     String?
  memberToken   String       @unique
  seatedSeatId  String?
  readyAt       DateTime?
  lastSeenAt    DateTime     @default(now())
  leftAt        DateTime?
  createdAt     DateTime     @default(now())
  updatedAt     DateTime     @updatedAt

  @@index([roomId, lastSeenAt])
  @@index([profileId])
}
```

如果第一版不想新增成员表，可以先简化为“用户填写昵称后直接选择座位”。但这会弱化你要求的房间成员区体验。

### 6.6 玩家战绩

为了支持个人主页统计，建议新增通用桌游记录模型：

```prisma
model GameToolPlayerRecord {
  id            String       @id @default(cuid())
  roomId        String
  profileId     String
  kind          GameToolKind
  variantKey    String
  seatNumber    Int?
  roleKey       String?
  alignment     String?
  result        String       // WIN / LOSE / JUDGE
  wasJudge      Boolean      @default(false)
  playedAt      DateTime
  createdAt     DateTime     @default(now())

  @@unique([roomId, profileId])
  @@index([profileId, playedAt])
  @@index([kind, variantKey, playedAt])
}
```

## 8. 状态机

```text
LOBBY
  -> 成员进入、选座、换座
  -> 全员落座并准备

READY
  -> 法官可以开始游戏

DEALING
  -> 系统锁房、洗牌、写入角色

IN_PROGRESS
  -> 玩家查看身份
  -> 法官主持线下流程
  -> 法官标记死亡 / 复活
  -> 法官结束游戏

FINISHED
  -> 展示胜负动画
  -> 写入玩家记录
  -> 返回已结束房间
```

关键服务端约束：

- 只有 `LOBBY` / `READY` 可以换座
- 只有全员落座并准备才能进入 `READY`
- 只有法官能开始游戏
- 只有法官能标记死亡 / 复活
- 只有法官能结束游戏
- 发牌和写战绩必须幂等
- 任何公共页面不能泄露普通玩家不应看到的身份

1. 先合并入口、版型配置和空房间。
2. 再合并选座、准备和开始。
3. 再合并发牌和玩家私密页。
4. 最后合并法官死亡、结束游戏和记录。

## 12. 设计注意事项

- 狼人杀现场主要靠线下发言，工具不要把玩家注意力长期拉到手机上。
- 投票、计时、发言顺序、夜晚行动和技能使用都交给线下法官，不进入 MVP 系统流程。
- 玩家页要极简，只突出座位号、身份卡、防窥和死亡状态。
- 法官页才是操作密度最高的地方，但也要避免小按钮堆叠。
- 角色图和卡面应使用 Friemi 原创素材，不使用第三方 App 牌面或未经授权图片。
- 角色说明只写简短功能提示，不复制现成说明书。
- WebView 壳可以直接承载 MVP，但 CSS 必须按移动浏览器标准做，不只按 App 壳调。
- 个人战绩不要过早竞技化；第一版只做局数、胜负和法官次数。

## 13. 当前结论

狼人杀 MVP 适合放在 v2.3 做，并且适合先用移动 Web 实现。它能复用 Avalon 已经证明过的房间、座位、私密 token 和事件底座，同时比 Avalon 的自动规则闭环更轻：MVP 只需要发身份、法官控场、死亡标记、手动结算和个人记录。

第一版重点不是“完整狼人杀线上化”，而是让线下组局开局更快、身份更安全、法官管理更省心。
