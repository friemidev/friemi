# Friemi README v2.4 产品介绍改版方案

> 分支：`docs/readme-v2-4-product-overview`  
> 目标：把根目录 `README.md` 从旧版功能清单，升级成“面向用户、协作者、审核者和新成员”的完整产品介绍页。  
> 当前阶段：先做统筹分析与落地方案，待复核后再正式改 README 和截图资产。

---

## 1. 改版目标

当前 README 的问题不是信息太少，而是讲法偏工程和版本清单：

- 用户看不出 Friemi 打开后到底长什么样。
- 新加入的协作者很难理解“大厅 / 组局 / 活动 / 足迹 / 桌游工具 / 个人主页”的关系。
- 移动端 v2.3/v2.4 的 UI 已经发生较大变化，但 README 仍像 v1/v2.1 的产品文档。
- README 里缺少真实页面截图，外部读者无法快速建立产品印象。
- 组局、活动、Moment 足迹、消息、桌游工具、WebView / App 形态没有被串成一个完整产品故事。

新版 README 需要回答三个问题：

1. Friemi 是什么？
2. 用户打开后能做什么？
3. 这个项目现在有哪些端、功能和技术结构？

## 2. README 目标读者

| 读者 | 关心内容 | README 需要给出的信息 |
| --- | --- | --- |
| 普通用户 | 这个产品有什么用，怎么使用 | 大厅、发现活动、创建组局、报名、足迹、消息、个人主页截图 |
| 合作者 / 同事 | 当前产品边界和页面结构 | 核心模块、路由、截图资产、开发入口 |
| App 审核 / 商店资料准备 | 产品形态和功能说明 | 用户账号、组局、消息、通知、内容安全、隐私 |
| 新开发者 | 如何启动项目、主要目录在哪 | 技术栈、项目结构、本地开发、数据库和发布注意事项 |
| 潜在运营 / 主理人 | 如何发起活动和传播 | 创建组局、分享海报、报名管理、个人主页 |

## 3. 新 README 推荐结构

### 3.1 顶部 Hero

建议保留品牌图，但替换文案为更清楚的产品定义：

```text
Friemi
在城市里发现活动，发起组局，留下足迹。
```

顶部链接建议更新为：

- 官网：`https://www.friemi.com`
- 更新日志：`apps/web/app/[locale]/updates` 或 `docs/v2_4`
- 产品截图：README 内部锚点
- 本地开发：README 内部锚点

### 3.2 一句话定位

推荐文案方向：

```text
Friemi 是一个面向海外生活场景的活动发现与组局平台。
用户可以发现城市活动、发起自己的组局、报名参加朋友的计划、用足迹记录日常，并在移动端 / WebView / 浏览器中保持一致体验。
```

避免：

- “MVP”
- “管理后台”
- “数据库能力”
- “套壳”
- 过多技术词作为产品介绍

### 3.3 产品总览截图墙

README 开头应放一个 2x3 或横向截图墙，让用户先看到产品。

### 3.4 用户可以做什么

建议分成 6 个产品模块，每个模块配 1 张图或 1 行截图。

#### 大厅：打开就知道今天可以做什么

覆盖：

- 城市选择
- 搜索活动和用户
- 热门分类
- 最新动态 / 推荐活动
- 移动端底部导航

#### 组局：把一个想法变成可报名页面

覆盖：

- 组局大厅
- 组局详情
- 我要报名 / 取消报名
- 报名人数、时间地点、发起人
- 分享入口

#### 创建组局：轻量发布，不像后台表单

覆盖：

- 上传封面
- 类型选择
- 城市和地点
- 开始 / 结束时间
- 人数上限、费用、外部链接
- 审核开关

#### 活动：发现城市里的真实活动

覆盖：

- 活动大厅 `/activities`
- 活动详情 `/public-events/[publicEventId]`
- 从活动发起组局
- 外部详情 / 抢位置链接

#### 足迹：Moment、消息和主页放在一起

覆盖：

- 发布 Moment：文字 + 多图
- 点赞、评论、转发、举报
- 消息列表和聊天详情
- 用户主页摘要

#### 桌游工具：线下局的轻量助手

覆盖：

- `/game-tools`
- Avalon
- 狼人杀
- 房间、座位、身份卡、公共屏

### 3.5 Web / App / WebView 形态说明

README 需要清楚解释：

- Friemi Web 是同一套核心体验。
- Android / iOS 当前可以通过 WebView 承载同一网站。
- 移动端 UI 优先适配浏览器、微信 WebView、Android WebView、iOS Safari。
- 原生能力重点包括系统通知、App badge、登录回跳、WebView 安全区。

这个部分不要写成“套壳临时方案”，而应表达为：

```text
Friemi 的移动体验以 Web 为核心，App 端复用同一套产品能力，并逐步补齐系统通知、徽章和原生入口。
```

### 3.6 当前能力清单

建议改成用户视角，而不是工程视角：

| 模块 | 当前能力 |
| --- | --- |
| 活动发现 | 城市活动、分类、搜索、收藏、活动详情 |
| 组局 | 创建组局、公开 / 私密、报名审核、取消报名、分享 |
| 好友与消息 | 好友码、好友申请、私聊、图片消息、未读数 |
| 足迹 | 图文动态、点赞、评论、转发、举报、通知 |
| 个人主页 | 发起、参与、收藏、好友、资料管理 |
| 桌游工具 | Avalon、狼人杀、座位、发身份、公共屏、复盘 |
| 多语言 | 中文、英文、法语 |
| App 能力 | Android / iOS WebView、通知、Badge、登录回跳 |

### 3.7 技术与开发

保留但压缩：

- 技术栈
- 项目结构
- 本地开发

- 质量检查

### 4.2 截图尺寸

建议先使用统一移动端截图，后续再补桌面截图：

```text
390 x 844   iPhone 12/13 常见宽度
430 x 932   大屏 Android / iPhone Pro Max 近似
1440 x 1000 桌面总览截图，可选
```

README 第一版建议主要放 390px 移动截图，避免横向图太多。

### 4.3 截图数据要求

截图不能包含：

- 真实邮箱、手机号、微信号
- 真实聊天隐私
- 真实用户头像如未授权
- 暴露后台 / 管理入口
- 预览数据库敏感信息

推荐方式：

1. 使用预览环境或本地环境。
2. 建一个 demo 用户和 demo 组局。
3. 使用公开活动和公开组局作为截图内容。
4. 聊天截图如需展示，使用 demo 对话或打码。
5. 所有截图统一存入 `apps/web/public/readme/v2_4/`。

### 4.4 截图采集方式

优先使用 Playwright，保证尺寸统一：

```text
scripts/capture-readme-screenshots.ts
```

建议脚本能力：

- 读取 `README_SCREENSHOT_BASE_URL`
- 固定 viewport
- 自动访问页面
- 等待关键 selector
- 保存 PNG 到 `apps/web/public/readme/v2_4/`
- 可选隐藏 Next/Vercel dev badge

如果页面需要登录，第一版可以先手动截图，后续再补登录态脚本。

## 7. 风险与注意事项

- README 截图一旦使用真实线上数据，需要确认没有隐私信息。
- 移动端截图应避免出现浏览器调试边框、微信翻译条、Vercel preview 域名。
- 如果用本地截图，数据应足够真实，不要出现大量 `test2`、空头像、占位文案。
- README 不要写“目前没人下载”“临时套壳”等内部措辞。
- 不要把所有功能都吹成已完成。比如桌游工具可以写“已接入 Avalon / 狼人杀工具”，但血染钟楼应明确是预留或敬请期待。
- 不要把工程命令写在 README 开头，避免普通用户一打开就看到开发信息。
