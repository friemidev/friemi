# 狼人杀房间无痛同步优化 Checklist

> 适用范围：`/game-tools/werewolf/rooms/{roomId}`
> 目标：保留轻量刷新同步，但让用户感知不到“页面闪动、跳转、状态弹回去”。

## 1. 产品边界

狼人杀仍然是线下辅助工具，不做完整线上实时对局。这里的同步目标不是毫秒级竞技同步，而是让线下开局时的入座、离座、准备状态足够快、足够稳、足够不打扰。

判断标准：

- 用户点入座、离座、准备后，本机立刻看到结果。
- 其他玩家不用手动刷新，几秒内能看到变化。
- 自动刷新不打断当前操作，不出现明显白屏、闪屏、回弹。
- 服务端仍然做最终校验，座位被抢、网络失败等情况可以回到真实状态。

## 3. 当前实现策略

### 本机操作

入座、离座、准备按钮提交前，先更新当前浏览器里的房间状态。用户看到的是立即反馈，而不是等待服务端返回。

服务端动作仍然写入数据库、记录事件、刷新缓存。区别是房间页传入 `responseMode=inline` 后，服务端成功时返回状态，不再跳转页面。

### 其他玩家同步

房间页在前台时使用轻量短轮询。轮询先请求房间同步接口，只比较版本号；版本没变不做任何 UI 更新，版本变了才请求房间快照并合并到本地状态。

- 候场：约 1.8 秒探测一次。
- 游戏中：约 4.2 秒探测一次。
- 低端设备：候场约 2.8 秒，游戏中约 6.2 秒。
- 慢网络 / 省流模式：候场约 4.2 秒，游戏中约 8.2 秒。
- 页面隐藏时不主动刷新。
- 回到前台、重新联网时立即探测，有变化才刷新。
- 同一浏览器多标签页之间通过 `BroadcastChannel` 主动通知，减少等待下一次轮询的时间。

这个方案不依赖 WebSocket，部署和维护成本低，也符合线下辅助工具的定位。

### 防闪动

本地刚做完操作后，会进入短暂保护窗口。在保护窗口内，自动刷新不会把界面覆盖成旧数据。服务端确认成功后，会拉取房间快照并合并；服务端失败后，会强制回到真实状态，并显示对应错误。

## 5. 后续建议

### 暂缓 Supabase Realtime / WebSocket

当前狼人杀定位仍是线下辅助工具，候场入座和准备状态不需要毫秒级推送。引入 Realtime 会增加连接管理、鉴权、掉线恢复和移动端省电问题。只有当真实多人测试确认“几秒内同步”仍然不够用时，再升级。

### 暂缓离线操作队列

入座、离座、准备都属于强一致操作，离线排队可能导致恢复网络后抢座、换座、准备状态过期，反而更容易误导玩家。当前更合适的策略是断网时阻止操作，恢复网络后重新同步。

### 暂缓字段级 patch

当前已经从整页刷新推进到房间 JSON 快照合并。字段级 patch 可以进一步减少数据量，但会增加冲突合并复杂度。房间规模很小，现阶段 JSON 快照足够轻，优先保留简单可靠的方案。
