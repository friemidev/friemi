# v2.0 功能实现清单

## 当前阶段目标

v2.0 是项目搬迁后的第一个大版本，重点不是继续堆功能，而是完成项目转移后的品牌重建、全站视觉风格统一、关键体验缺陷修复和发布前测试体系补齐。

当前主线：

```text
项目完成迁移 -> 品牌从旧项目过渡到 Friemi -> 全站 UI 有统一腔调 -> 核心路径更稳定 -> 预览和生产发布可被测试验证
```

## 分支规则

所有功能从 `dev` 拉独立 feature 分支：

```bash
git switch dev
git pull friemi dev
git switch -c feature/功能名
```

完成后 PR 到：

```text
feature/* -> dev
```

v2.0 不建议把品牌替换、页面重构、bug 修复和测试体系放在同一个 PR。每个分支只解决一个明确问题，方便预览、回滚和发布验证。

## P0：v2.0 必须优先完成的迁移与品牌基础

### 1. 项目迁移收尾与部署归属确认

建议分支：

```text
feature/v2-project-migration-hardening
```

需求理解：

项目已经从旧仓库 / 旧部署关系迁出，但 v2.0 发布前还需要确认 GitHub remote、Vercel project、Supabase Preview / Production、Storage、Clerk 和 cron 都指向新项目，避免出现代码推到新仓库但部署仍连旧资源的情况。

小功能：

- [x] 确认 `dev` 默认推送到 `friemi/dev`，不再误推旧 `origin`
- [x] 确认 Vercel Preview / Production 绑定的是新项目
- [x] 确认 Vercel Preview / Production 的 `DATABASE_URL` 和 `DIRECT_URL` 分别指向新预览库和新生产库
- [x] 确认 Supabase Storage bucket `activity-covers` 在新预览和新生产都可读写
- [x] 确认当前阶段 Clerk 使用策略：继续沿用旧 Development Clerk，或切换到新 Development Clerk 后完成关键账号重绑
- [ ] 确认 cron 手动调用和 Vercel 定时触发都能写入新生产库
- [ ] 清理旧项目引用、旧域名、旧 repo 名和旧部署说明
- [ ] 更新项目迁移 runbook 中已完成项和剩余风险

验收标准：

- `git push` 默认不会推到旧仓库
- `vercel` 和 `vercel --prod` 部署到正确项目
- 预览环境和生产环境不会混用数据库
- 上传封面、公共活动导入、登录、报名和消息都写入新资源
- 迁移相关风险在文档中有明确状态

- [x] 检查旧品牌、旧仓库名、旧域名等展示文案，区分需要替换和历史兼容引用
- [x] 保留必要的旧品牌兼容字段，避免 URL、数据库 source 或历史内容被误删

### 3. 全站设计语言重构

建议分支：

```text
feature/v2-design-system-refresh
```

需求理解：

v1.x 的页面已经完成大量功能，但视觉语言是逐步堆出来的，存在按钮、卡片、chip、空状态、加载态、筛选栏、分页和详情页风格不完全统一的问题。v2.0 需要建立一套更有识别度、更克制、更适合活动 / 组局产品的视觉基础，而不是每个页面单独美化。

小功能：

- [ ] 定义 v2.0 色彩系统：主色、辅助色、成功、警告、错误、已结束、禁用、背景和边框
- [ ] 定义按钮层级：主按钮、次按钮、幽灵按钮、危险按钮、已报名 / 管理 / 查看状态按钮
- [ ] 统一卡片圆角、边框、阴影、背景和 hover / active 状态
- [ ] 统一活动卡片和组局卡片的视觉差异，保留语义清楚但不突兀
- [ ] 统一 badge / chip 风格：活动、组局、我发起、已报名、已结束、进行中、开放局
- [ ] 统一输入框、select、筛选栏、搜索框和分页组件
- [ ] 统一全站空状态、加载 skeleton、错误提示和 toast
- [ ] 增加轻量触控反馈，移动端点击不生硬但不影响性能
- [ ] 避免单一色系堆叠，避免页面读起来像临时模板

验收标准：

- 用户能明显感到 v2.0 是同一套产品风格
- 活动、组局、自己发起、已结束、已报名等状态语义清楚
- 桌面端 1280px 和移动端 390px 都没有横向滚动条
- 主要按钮触控高度不低于 44px
- 不出现文字被底部导航、徽章、收藏按钮或浮层遮挡
