## 设计读法

Reading this as：移动优先的真实活动 / 组局社交产品，面向想快速发现活动、加入组局、联系朋友和回看个人动态的用户；视觉语言应当温暖、可信、轻快、有生活感，但不能变成营销落地页，也不能牺牲列表浏览效率。

建议参数：

```text
DESIGN_VARIANCE: 5
MOTION_INTENSITY: 3
VISUAL_DENSITY: 6
```

原因：

- Next Fun 是一个高频浏览和操作产品，不是纯展示型 landing page，因此不能追求过强的实验性排版
- 移动端空间紧张，需要提升信息密度，但仍要避免拥挤、卡片重叠和横向溢出
- 现有产品已经形成暖色纸张、珊瑚色主按钮、浅绿状态、圆角卡片的视觉基础，优化应当在这个基础上收敛，而不是重做成另一套风格
- 动效只用于状态反馈、页面进入、按钮反馈和轻量展开，不做大面积滚动特效

## 使用原则

- [ ] 先审查现有页面，再决定改法；不直接套用通用模板
- [ ] 保留现有品牌识别：Next Fun logo、暖色背景、珊瑚主色、活动 / 组局卡片差异
- [ ] 移动端优先处理信息层级、按钮位置、首屏密度和底部导航遮挡
- [ ] 网页端优先处理内容最大宽度、左右留白过大、模块分散和空状态
- [ ] 活动、组局、个人空间、搜索、详情页要使用同一套卡片、徽章、按钮和空状态语言
- [ ] 后台页面可以更工具化，不强行做成活动页风格，但也不能横向溢出或信息卡过大
- [ ] 不引入 AI 紫蓝渐变、玻璃拟态大面板、装饰性大光斑和无意义动效
- [ ] 不为了“高级感”降低可读性、触控面积或列表扫描效率
- [ ] 每个实施步骤都要补充移动端与网页端截图验证点

#### 颜色规则

| 用途 | 当前基准 | 使用规则 |
| --- | --- | --- |
| 页面底色 | `#f5efe2` / `paper #f7f3ea` | 全站保留暖纸张底色，不切到纯白或冷灰大背景 |
| 主文字 | `ink #151515` | 标题、重要数字、主导航使用；不要用过浅灰色做关键文字 |
| 主行动 | `#d88d72` / hover `#c87b61` | 报名、搜索、发起组局等主 CTA 使用；避免再新增绿色主按钮 |
| 次行动 | `white/80` + `ring black/10` 或暖边框 | 查看、返回、取消、联系等操作使用轻按钮，不抢主 CTA |
| 社交 / 关系 | `moss #556b4e` / `moss/10` | 好友、关注、申请成功、社交信号使用；不用于报名主按钮 |
| 组局卡片 | `#fffaf4` + `#e1b89c` / `#d88d72` | 组局用暖色体系，强调“人和参与” |
| 活动卡片 | `#f8fdff` + `#b9d7e5` / `#7eb7cf` | 活动信息用浅蓝体系，强调“信息和发现” |
| 已结束 / 不可操作 | `zinc-50` / `zinc-300` / `zinc-500` | 灰色只表达结束、取消、不可用；不要用灰色做普通活动按钮 |
| 分割与边框 | `#d8cdbb` / `#eadfce` / `#dfccb2` | 用细边框和 ring 表达层级，少用重阴影 |
| 提醒 / 错误 | red / amber 系统色 | 仅用于错误、警告、危险操作，不混入品牌主视觉 |

后续代码收敛原则：

- 优先复用 Tailwind 已有 `ink / paper / moss / clay / sky`
- `#d88d72` 是实际主 CTA 色，后续改代码时可以逐步收敛成语义 token，例如 `coral`
- 活动浅蓝、组局暖橙、已结束灰色是核心语义，不要在页面之间互换
- 直接写十六进制色值目前集中在卡片、筛选、分页、详情页和搜索页；后续改这些页面时同步减少重复色值

#### 圆角规则

| 元素 | 建议半径 | 说明 |
| --- | --- | --- |
| 页面大模块 | `rounded-[1.25rem]` 或 `rounded-2xl` | 只用于筛选区、详情侧栏、弹窗主体、空状态 |
| 内容卡片 | `rounded-lg` 到 `rounded-xl` | 活动 / 组局卡片、后台小卡、消息卡使用，不继续增大 |
| 图片封面 | 跟随卡片裁切 | 卡片图片不单独加更大圆角，避免层级混乱 |
| 按钮 / chip / tab | `rounded-full` | 主要操作和筛选 chip 使用 pill，形成 Next Fun 的轻快感 |
| 输入框 | `rounded-xl` 或 `rounded-full` | 搜索框可 pill；普通表单输入用 `rounded-xl` |
| 底部导航项 | `rounded-2xl` | 保持触控友好和当前选中反馈 |

禁止项：

- 同一个组件内不要混用 `rounded-md`、`rounded-2xl`、`rounded-full`，除非有明确语义差异
- 移动端不要用过大的外层圆角盒子包住整个列表，会降低内容密度

#### 阴影与层级规则

| 层级 | 使用场景 | 规则 |
| --- | --- | --- |
| 0 | 页面主体、普通分区 | 只用背景和间距，不加阴影 |
| 1 | 普通卡片、空状态、小面板 | `shadow-sm` 或低透明暖色阴影 |
| 2 | 可点击活动 / 组局卡片 | 轻微 hover 位移和柔和阴影，移动端不依赖 hover |
| 3 | header、底部导航、popover | 允许方向性阴影，例如底部导航向上投影 |
| 4 | 全屏弹窗、确认弹窗 | `shadow-2xl`，但背景要简洁，避免玻璃拟态 |

规则：

- 不使用纯黑重阴影；阴影要带暖色或低透明度
- 列表卡片不做强浮起，否则移动端会显得拥挤
- 弹窗和 header 可以有明显层级，但不能遮挡主要 CTA

#### Badge 与状态规则

| 状态 | 位置 | 视觉 |
| --- | --- | --- |
| 活动 | 卡片图片左上角 | 浅蓝底、深蓝字、小圆点，必须醒目但不压图片标题 |
| 组局 | 卡片图片左上角 | 珊瑚底、白字、小圆点，作为大类标识 |
| 分类 | 图片左下角 | 深色半透明小矩形，表达展览 / 戏剧等内容分类 |
| 开放局 / 私人局 | 图片左下角分类旁 | 暖白小徽章，只在组局卡片显示 |
| 进行中 / 未开始 / 已结束 | 图片右下角 | 小白底状态；已结束可灰度化 |
| 我发起 | 图片右上角 | 暖白 + crown，不和收藏爱心同时争抢位置 |
| 收藏爱心 | 图片右上角 | 自己发起的卡片不显示；普通卡片显示 |
| 已报名 / 待确认 | 主按钮区域 | 不用绿色，使用暖色或琥珀轻状态 |
| 管理组局 / 查看活动 | 主按钮区域 | 自己发起内容用中性暖白按钮，不误导为报名 |

#### 移动端密度规则

- 页面左右安全边距：常规页面 `px-4`，极窄内容可以 `px-3`
- Header 固定 64px，高度不要再增；移动端滚动进度条在 header 下方
- 底部导航高度约 `5.15rem`，页面底部内容需要 `pb-32` 左右避让
- 列表卡片 gap：双列卡片用 `gap-3`，单列卡片用 `gap-4`
- 双列卡片内标题最多 2 行，时间可 2 行，地点 1 行，避免撑爆卡片
- 筛选区默认紧凑，移动端不要用大外框包两层筛选
- 空状态不要超过半屏；优先短文案 + 一个行动

#### 网页端布局规则

- 常规页面默认 `max-w-7xl`
- 个人空间、搜索结果、活动大厅、组队大厅需要利用 2 到 4 列内容，不让内容只贴左侧
- 详情页正文建议 `max-w-3xl`，侧栏 `320px` 左右；移动端侧栏前置时必须压缩
- 后台页可以突破 `max-w-7xl` 到 `92rem / 96rem / 110rem`，但移动端必须 `overflow-x-hidden`
- 表单页保持 `max-w-3xl`，避免输入区过宽

#### 后续页面审查清单

- [ ] 这个页面的活动 / 组局 / 已结束语义是否和卡片体系一致
- [ ] 主按钮是否只有一个，颜色是否使用珊瑚主色
- [ ] 次按钮是否使用白底细边框，而不是另一个强色按钮
- [ ] 移动端首屏是否能看到核心内容，而不是只看到说明文案或登录框
- [ ] 移动端是否存在横向滚动条
- [ ] 底部导航是否遮挡按钮、分页、输入框或 toast
- [ ] 空状态是否给了下一步行动
- [ ] 网页端是否存在内容过窄、过空或偏左的问题
- [ ] hover / focus / active / disabled 状态是否完整
- [ ] 文案是否减少解释，把空间留给内容和操作

#### 组件所有权

这些组件是后续 taste-skill 优化的“源头组件”。优先修源头，不在页面里复制一套新样式。

| 组件 | 角色 | 后续处理规则 |
| --- | --- | --- |
| `ActivityCard` | 活动 / 组局卡片核心 | 所有列表页优先复用它；新增卡片样式必须先判断是否应该扩展这个组件 |
| `ActivityCardMasonryGrid` | 搜索页移动端错列布局 | 只用于搜索结果这类发现流；首页、大厅、活动页默认不使用瀑布流 |
| `ActivityFilters` | 活动页筛选入口 | 保持轻量筛选栏，不回到大表单框；下拉筛选即时生效 |
| `ActivityLobbyView` | 组队大厅筛选和分页 | 筛选 chip 不能横向滚动，移动端不加大外框 |
| `PaginationControl` | 分页统一控件 | 活动页、组队大厅复用；不能在页面内重新写独立分页样式 |
| `EmptyState` | 通用空状态 | 普通空状态使用它；局部紧凑空状态可以扩展，但结构要一致 |
| `AppHeader` / `DesktopNav` / `MobileNav` | 全局导航 | 页面不单独做 header；当前页反馈、进度条、底部导航都从这里收敛 |
| `ProfileDashboardView` / `ProfileActivitySections` | 个人空间结构 | 个人资料、社交数据、内容 tab 的密度规则从这里收敛 |
| `packages/ui` | 基础 UI primitives | 目前偏中性 zinc 风格；后续需要逐步贴近 Next Fun token，而不是各页面重复覆盖 |

组件修改顺序：

1. 先改 `packages/ui` 或高复用组件的默认能力
2. 再改页面级布局和间距
3. 最后处理特殊页面例外

例外规则：

- 后台页面可以不完全使用活动 / 组局卡片语义，但必须使用同一套按钮、圆角和空状态规则
- 详情页可以有更强的 hero，但移动端第一屏必须优先信息密度和转化，不追求视觉大图占满

#### Token 迁移建议

现状：Tailwind 已有 `ink / paper / moss / clay / sky`，但大量实际使用色仍写在 class 中。后续不需要一次性重构所有颜色，应该按页面改动顺手迁移。

建议扩展 token：

| 新 token | 建议色值 | 作用 |
| --- | --- | --- |
| `coral` | `#d88d72` | 主 CTA、当前页进度、主要选中态 |
| `coral-dark` | `#c87b61` | 主 CTA hover / active |
| `coral-soft` | `#fff2e7` | 已报名、轻提示、暖色 chip 背景 |
| `sand` | `#eadfce` | 分割线、分页轨道、轻边框 |
| `sand-strong` | `#dfccb2` | 按钮 ring、筛选 chip 边框 |
| `team-bg` | `#fffaf4` | 组局卡片底色 |
| `team-border` | `#e1b89c` | 组局卡片边框 |
| `event-bg` | `#f8fdff` | 活动卡片底色 |
| `event-border` | `#b9d7e5` | 活动卡片边框 |
| `event-accent` | `#7eb7cf` | 活动卡片顶部条和活动 badge |

迁移节奏：

- 第一次真正改代码时，优先把 `#d88d72 / #c87b61 / #eadfce / #dfccb2` 迁进 Tailwind config
- 卡片语义色等到改 `ActivityCard` 时一起迁，不单独做机械重构
- `packages/ui` 的 `Button primary` 目前是 `zinc-950`，后续需要区分“品牌主按钮”和“后台 / 工具黑按钮”，不要直接把所有 primary 改珊瑚

#### 组件配方

后续写 UI 时优先按这些配方组合，不重新发明样式。

主 CTA：

```text
h-10 或 h-11 + rounded-full + bg-coral + text-white + hover:bg-coral-dark + 轻暖色阴影
```

次 CTA：

```text
h-9 或 h-10 + rounded-full + bg-white/80 + text-ink 或 text-[#5b4b3a] + ring-1 ring-sand-strong
```

筛选 chip：

```text
rounded-full + px-3/4 + h-9/10 + bg-white/75 + ring-1 ring-sand + active:bg-moss/10 或 active:bg-coral-soft
```

内容卡片：

```text
rounded-lg/xl + border/ring + bg-white/75 或语义底色 + shadow-sm/低透明暖阴影
```

空状态：

```text
border-dashed + bg-white/55-70 + icon circle + title + 最多一行说明 + 一个行动
```

弹窗：

```text
移动端全屏或底部 sheet；桌面 rounded-[1.5rem] max-w；bg-paper；shadow-2xl；头部固定，内容滚动
```

#### 页面风险矩阵

| 页面 / 区域 | 当前风险 | 优先级 | 第一判断 |
| --- | --- | --- | --- |
| 首页 `/home` | hero 与最近活动之间风格容易割裂；CTA 可能像普通按钮 | 高 | 需要和大厅 / 活动入口建立同一套“发现 + 前往”语言 |
| 组队大厅 `/lobby` | 筛选项多，移动端容易占首屏；活动和组局混合时语义必须清楚 | 高 | 保持双列和紧凑筛选，不再加外框 |
| 活动大厅 `/activities` | 筛选区和列表之间容易分离；分页容易笨重 | 高 | 筛选要像工具栏，分页要像列表尾部控制 |
| 搜索 `/search` | 内容类型多、相关度层级复杂；最容易出现认知混乱 | 高 | 搜索是最适合 taste-skill 的发现流，需要优先做精 |
| 活动 / 组局详情 | 移动端第一屏转化压力大；登录提示容易喧宾夺主 | 高 | 信息先于登录，主 CTA 只保留一个 |
| 个人空间 `/profile` | 网页端易空，移动端资料区易过高 | 高 | 顶部身份压缩，内容 tab 作为主结构 |
| 消息 / 好友 | 社交状态多，容易 badge 和按钮不统一 | 中 | 与 moss 社交语义绑定 |
| 后台 analytics / reports / scraper | 移动端卡片大、图表可能溢出 | 中 | 工具化、紧凑、无横向滚动 |
| 登录 / 微信引导 | 容易像外部系统页 | 中 | 保持 paper 背景和单主行动 |
| 商家 / updates | 使用频率较低但也会暴露卡片风格不一致 | 低 | 后续顺手对齐 |

#### 代码落地顺序

后续真正改 UI 时按这个顺序推进：

1. **Token 先行**：补 Tailwind semantic token，避免新增更多硬编码色
2. **源头组件**：`ActivityCard`、`PaginationControl`、`EmptyState`、`MobileNav`
3. **列表入口**：首页、组队大厅、活动大厅
4. **发现流**：搜索结果页
5. **转化页**：活动 / 组局 / 公开活动详情
6. **个人空间和社交弹窗**
7. **后台移动端**
8. **加载态、空状态、动效统一**

每次提交前检查：

- 如果改的是卡片：搜索、首页、大厅、个人空间至少各看一处
- 如果改的是按钮：报名、查看活动、管理组局、分页、返回来源都要扫一遍
- 如果改的是导航：桌面 header、移动 header、底部导航、滚动进度条都要扫一遍
- 如果改的是空状态：筛选无结果、真实空列表、加载失败三种都要分清

#### 截图验证矩阵

后续每个视觉步骤至少覆盖：

| 视口 | 宽度 | 必看内容 |
| --- | --- | --- |
| 小手机 | `360px` | 是否单列回退、按钮文字是否被切、底部导航是否遮挡 |
| 常见手机 | `390px` | 双列卡片、筛选区、分页、详情页第一屏 |
| 大手机 | `430px` | 双列间距、空状态、全屏弹窗 |
| 平板 / 窄桌面 | `768px` | 是否出现尴尬两栏或过宽单列 |
| 桌面 | `1280px` | 内容是否太窄、是否偏左、卡片数量是否合理 |
| 宽屏 | `1440px+` | header 搜索、导航、内容最大宽度是否协调 |

#### Taste Skill 使用边界

适合使用 taste-skill 的地方：

- 页面整体节奏、留白、卡片质感、字体层级、按钮和状态收敛
- 搜索结果这种发现流页面
- 个人空间这种内容密度和空状态都很重要的页面
- 移动端第一屏重排

不适合过度使用的地方：

- 后台表格和数据看板，不做实验排版
- 多步骤表单，不牺牲明确性
- 聊天输入区和通知列表，不加装饰性动效
- 已建立语义的灰色结束态、活动蓝、组局暖色，不为了“新鲜”随意换

验收标准：

- [x] 全站主要 UI 元素有明确的颜色、圆角、阴影、间距规则
- [x] 后续页面优化能引用同一套规则，不需要每个页面重新猜样式
- [x] 不改变业务逻辑，只建立设计约束和审查依据

### 3. 搜索结果页视觉与交互精修
